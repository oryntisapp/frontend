## 2. Tech Stack Additions (on top of the existing Vite/React/Tailwind/Framer Motion stack)

- **Authentication & user data:** Firebase — `firebase/auth` for sign-in/sign-up/session state, `firebase/firestore` (or a lightweight alternative if you prefer) for storing user profile fields Firebase Auth doesn't natively hold (company name, role, onboarding status), `firebase/storage` only if avatar upload is in scope.
- **Config:** a single `src/lib/firebase.ts` module that reads all config values from Vite env vars (`import.meta.env.VITE_FIREBASE_API_KEY`, `VITE_FIREBASE_AUTH_DOMAIN`, `VITE_FIREBASE_PROJECT_ID`, `VITE_FIREBASE_APP_ID`, etc.) — never hardcode the config object inline, even though these particular values are safe to expose client-side; keeping them in `.env` is still correct practice for environment management (dev/staging/prod projects) and keeps the repo portable. Add a `.env.example` with empty placeholders so it's obvious what needs to be filled in.
- **Routing:** React Router, with a clear split between a public route tree (`/login`, `/signup`, `/forgot-password`) and a protected route tree (everything under `/dashboard/*`).
- **Forms:** same lightweight validation approach as the landing page — plain controlled state or `zod` if it's already a dependency; no need for a heavy form library for the handful of forms this needs (login, signup, forgot password, profile edit, change password).

## 5. Firebase Authentication — Full Specification

### 5.1 Auth methods required
- **Email/password** sign-up and sign-in.
- **Google sign-in** via `GoogleAuthProvider` + `signInWithPopup` (with a `signInWithRedirect` fallback path for browsers/webviews that block popups — detect the popup-blocked error and fall back automatically rather than just failing silently).
- **Forgot password** via `sendPasswordResetEmail`.
- **Session persistence** via Firebase's default `onAuthStateChanged` listener, wrapped in a single `AuthContext`/`AuthProvider` at the app root exposing `{ user, loading, signIn, signUp, signInWithGoogle, signOut, sendPasswordReset }` — every screen reads auth state from this context, nothing re-implements its own Firebase calls.

### 5.2 Auth screen requirements
- `/login`: email + password fields (with a show/hide password toggle), `Sign In` button, `Continue with Google` button (secondary style, Google's standard "G" mark, not a generic OAuth icon — brand guidelines matter here, don't reskin Google's logo), a `Forgot password?` link, and a `Don't have an account? Sign up` link. Real inline validation (invalid email format, empty fields) and real Firebase error mapping to plain-English messages (e.g. `auth/wrong-password` / `auth/user-not-found` → "That email or password doesn't look right," not the raw Firebase error code).
- `/signup`: name + email + password + confirm-password, same Google option, real password-strength/match validation, and — since this maps to the BA scope's pricing tiers — an optional pre-selected plan carried in via query param (e.g. arriving from a pricing page's "Start on Growth" button) so signup can say `Signing up for the Growth plan` instead of being generic.
- `/forgot-password`: single email field, sends the reset email, shows a clear "check your inbox" confirmation state instead of silently succeeding with no feedback.
- All three screens share the same visual treatment: a centered glass card on the existing ambient-gradient background (reuse the landing page's `AmbientBlobs`/gradient-orb background system so auth pages don't look like a different, unbranded product), Oryntis logo above the card, a link back to the marketing site beneath it.

### 5.3 Profile management (`/dashboard/settings/profile`)
- Editable display name, avatar (Google photo if signed in via Google, otherwise an initials avatar with an optional upload if Firebase Storage is in scope), and email address.
- **Email changes require re-authentication** — Firebase requires a recent sign-in for sensitive account changes; prompt the user to re-enter their password (or re-trigger the Google popup) before calling `verifyBeforeUpdateEmail`, and show a clear "why am I being asked this" microcopy line so it doesn't feel like a bug.
- Save state feedback: a real success/error toast or inline confirmation on save, not a silent no-op.

### 5.4 Security settings (`/dashboard/settings/security`)
- **Change password** — only shown/enabled if the user actually has the `password` provider linked (a Google-only account has no password to change). For password-provider users: current password + new password + confirm, requiring re-authentication via `reauthenticateWithCredential` before calling `updatePassword` — Firebase will reject the update otherwise, so build the re-auth step in from the start rather than discovering the requirement via a runtime error later.
- **Connected accounts** — show Google as either "Connected" (with an `Unlink` action, disabled if it's the user's *only* sign-in method, since unlinking their last provider would lock them out — show a clear explanation if disabled) or "Not connected" (with a `Connect Google` action using `linkWithPopup` to add it to an existing email/password account).
- **Danger zone** — `Delete Account`, behind a real confirmation modal (type the word "delete" or re-enter password to confirm), calling Firebase's `deleteUser` after re-authentication.

### 5.5 Error handling patterns to build once, reuse everywhere
Write one small `mapFirebaseError(code: string): string` utility covering the common codes (`auth/email-already-in-use`, `auth/weak-password`, `auth/wrong-password`, `auth/user-not-found`, `auth/popup-closed-by-user`, `auth/account-exists-with-different-credential`, `auth/requires-recent-login`, `auth/too-many-requests`) and use it in every auth-related form's error state — don't let raw Firebase error strings reach the UI anywhere.

## 6. Functional Requirements (Carried Forward, Now Applied to the Dashboard)

Every rule from the landing page's functional addendum applies here too, with dashboard-specific emphasis:

- **Every sidebar item, topbar control, widget CTA, and settings action must perform a real action.** No decorative "Connect a system" button that doesn't open a real (even if mocked) connect flow; no notification bell that doesn't actually open a panel; no avatar dropdown item that doesn't navigate or execute (Sign Out must actually call Firebase's `signOut` and redirect to `/login`, not just close a menu).
- **Search must actually filter.** If it's not wired to real filtering logic yet, don't ship it as a static input — either wire it to client-side filtering against loaded data, or don't include it in this build pass.
- **Every settings save button must show a real result** (success, or a real error state) — this is doubly important on auth-adjacent forms, since a silent failure on "change password" is a trust-destroying bug, not a cosmetic one.
- **The onboarding checklist's `Continue Setup` button must route to the actual next incomplete step**, not always to the same place.

## 7. `.htaccess` for SPA Routing (Same Requirement as the Landing Page, Restated)

Because this dashboard adds several new deep routes (`/dashboard/settings/security`, etc.), the SPA-routing `.htaccess` requirement matters even more here than on the marketing site — a direct link to a nested settings page, or a password-reset email linking to `/login` with a query param, must survive a hard refresh. Place `.htaccess` in `/public` so it survives every build, with the same rules as before: rewrite non-file/non-directory requests to `index.html`, force HTTPS, long-lived caching on hashed assets, short/no-cache on `index.html`, correct MIME types. If this dashboard is deployed as a **separate subdomain or subpath** from the marketing site (e.g. `app.oryntisapp.com` vs `oryntisapp.com`), make sure `vite.config.ts`'s `base` and the `.htaccess` rewrite both match wherever it actually lives — test the same four-part check (direct load, client-nav, hard refresh, fresh-URL paste) against a genuinely nested route like `/dashboard/settings/security`, not just the root.

## 8. Final QA Checklist

1. Sign up with email/password, confirm the account actually appears in the Firebase console, and confirm the onboarding checklist starts at 0/4.
2. Sign in with Google on a fresh account, confirm the avatar and name populate correctly from the Google profile, and confirm signing out and back in with Google returns to the same account (not a duplicate).
3. Trigger "Forgot password," confirm the reset email arrives, and confirm the new password actually works afterward.
4. Attempt to change password on a Google-only account — confirm the UI correctly shows "set a password" (or hides the change-password form) rather than erroring.
5. Attempt to unlink Google when it's the only linked provider — confirm this is blocked with a clear explanation, not a silent failure or an account lockout.
6. Hard-refresh on `/dashboard/settings/security` directly — confirms both the auth persistence and the `.htaccess` routing fix are both working together correctly.
7. Log out — confirm every protected route immediately redirects to `/login`, and confirm `/login` itself redirects an already-logged-in user straight to `/dashboard`.
8. Full keyboard pass through the sidebar, topbar dropdown, and all three auth screens.
